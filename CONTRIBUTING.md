# Contributing

Thanks for helping with the community landing.

## Workflow

1. Open or pick an approved issue.
2. Create a branch named `type/description`.
3. Keep changes focused.
4. Open a PR using the template.
5. Wait for review before merging.

## Branch naming

Use one of these prefixes:

- `feat/`
- `fix/`
- `docs/`
- `refactor/`
- `chore/`
- `style/`
- `test/`
- `build/`
- `ci/`

Examples:

- `feat/hero-rewrite`
- `docs/community-rules`
- `chore/github-workflow`

## PR rules

- Link exactly one approved issue.
- Add exactly one `type:*` label.
- Keep the title clear and conventional.
- Explain the why, not just the what.
- Include screenshots when visual changes are involved.

## Content rules

- Keep public claims factual.
- Prefer short copy over dense paragraphs.
- Preserve bilingual support when touching user-facing text.
- Reuse the landing-context notes before introducing new ideas.

## Community expectations

- Be kind and direct.
- Review copy like product copy, not only code.
- Protect the shared direction; avoid one-off personal styles.
